## Что нужно сделать для автоматической установки картинок в VK

- Получить отдельный user access token от личного VK-аккаунта администратора сообщества `@factdaily`.
- Этот токен должен быть не ключом сообщества из `Управление -> Работа с API`, а OAuth-токеном пользователя-администратора.
- Нужные права user token:
  - `photos`;
  - `wall`;
  - `groups`;
  - `offline`.
- Получение user token:
  - зайти на `dev.vk.com` под личным аккаунтом администратора;
  - создать VK-приложение и получить его `client_id`;
  - открыть OAuth-ссылку вида:
    `https://oauth.vk.com/authorize?client_id=APP_ID&display=page&redirect_uri=https://oauth.vk.com/blank.html&scope=photos,wall,groups,offline&response_type=token&v=5.199`;
  - подтвердить доступ;
  - взять значение `access_token` из адресной строки после редиректа.
- Не присылать user token в чат и не записывать его в код.
- Добавить user token в Railway production как `VK_USER_ACCESS_TOKEN`.
- После этого доработать `app/vk_publisher.py`:
  - `VK_USER_ACCESS_TOKEN` использовать только для `photos.getWallUploadServer` и `photos.saveWallPhoto`;
  - `VK_ACCESS_TOKEN` сообщества оставить для `wall.post`;
  - передавать в `wall.post` attachment вида `photo<owner_id>_<photo_id>`.
- После доработки выполнить тест:
  - публикация VK без картинки;
  - публикация VK с картинкой;
  - полный цикл Telegram -> MAX -> VK -> Дзен.
- После успешного теста желательно перевыпустить текущий `VK_ACCESS_TOKEN`, так как он был передан в чат и один раз попал в локальный CLI-вывод при настройке Railway.

## Рекомендации по логам Telegram-токена

- Текущая проблема: `python-telegram-bot` отправляет запросы через `httpx`, а `httpx` на уровне `INFO` пишет полный Telegram Bot API URL.
- В этом URL токен находится прямо в пути:
  - `https://api.telegram.org/bot<TOKEN>/sendPhoto`;
  - `https://api.telegram.org/bot<TOKEN>/sendMessage`.
- Поэтому перевыпуск токена сам по себе не решает проблему: новый токен снова появится в логах при следующем `httpx` INFO-логе.
- Пока код не менять. Дождаться следующей плановой публикации в Telegram и Дзен, затем решить, нужно ли исправлять логирование.

Возможные варианты исправления, если решим делать:

- Самый быстрый вариант: понизить логирование `httpx` до `WARNING`, чтобы строки `HTTP Request: POST https://api.telegram.org/bot...` не попадали в Railway-логи.
- Дополнительно можно понизить логирование `telegram` и `telegram.ext` до `WARNING`, если эти логгеры начнут печатать лишнее.
- Более надежный вариант: добавить общий logging-фильтр, который маскирует `bot<TOKEN>` в любых сообщениях логов.
- Лучший практический вариант: совместить оба подхода:
  - `httpx` на `WARNING`;
  - маскирующий фильтр как страховка.
- Railway не сможет надежно скрыть токен, если приложение само печатает полный URL; исправлять нужно на уровне приложения.

  - если настройка не найдена — сохранять скриншот и список видимых контролов, но сначала лучше решить, должна ли публикация считаться failed.
- На первом этапе лучше тестировать на черновике или тестовой публикации, чтобы не ломать штатный выпуск.

Риски:

- Интерфейс Дзена может по-разному показывать настройку до публикации и после публикации.
- Если после публикации Дзен уводит на карточку материала, URL и доступные кнопки могут отличаться от редактора.
- Если настройка обсуждений меняется только в уже опубликованной статье, нужно будет добавить отдельный маршрут редактирования после публикации.
- Если изменение не сохранится, статья может остаться с обсуждением `Только подписчики`, даже если текст и картинка опубликованы корректно.

## Ручные действия

- В Дзене можно вручную удалить тестовые черновики с заголовками `Тест картинки Дзен — проверка загрузки`.
- В Дзене можно проверить опубликованную статью `Как устроен вирус`: она была создана автоматическим запуском после последних исправлений.
- В MAX можно удалить тестовые сообщения, если они мешают:
  - тест без картинки;
  - тест с картинкой;
  - первый ручной полный выпуск без картинки.
- Не публиковать и не пересылать значения токенов из `.env` и Railway-логов.
- Токен MAX был передан в чат во время подключения; после стабилизации желательно перевыпустить токен в MAX и заменить `MAX_BOT_TOKEN` в Railway.

## Текущие риски

- Интерфейс Дзена может измениться, поэтому Playwright-селекторы нужно проверять при новых сбоях.
- Если сессия Дзена устареет, нужно обновить `DZEN_STORAGE_STATE_JSON` в Railway.
- Railway-логи могут показывать URL Telegram Bot API с токеном; при подозрении на компрометацию токен лучше перевыпустить.
- Токен MAX был засвечен в чате подключения; при подозрении на компрометацию его лучше перевыпустить.
- Если Pexels/Pixabay временно не возвращают JSON или не находят картинку, Telegram, MAX и Дзен могут опубликовать статью без изображения.
- При локальном ручном запуске без предварительного `init_db()` публикация успела пройти, но запись статуса в локальную SQLite завершилась ошибкой `no such table: dzen_publications`; production FastAPI вызывает `init_db()` при старте.
